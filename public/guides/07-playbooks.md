# Playbook 管理功能指南

![Playbook 管理](/guides/images/playbooks_overview.webp)

Playbook 管理模块展示从 Git 仓库中自动扫描发现的所有 Ansible Playbook 文件。每个 Playbook 对应仓库中的一个 `.yml` 入口文件，以及其声明的变量配置。Playbook 是任务模板和自愈流程执行操作的**最底层执行单元**。

---

## 一、Playbook 状态 (status) 字典

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `pending` | **待配置** | Playbook 刚被扫描发现，变量配置尚未完成 🟡 |
| `ready` | **就绪** | 变量配置已确认，可以被任务模板引用和执行 🟢 |
| `error` | **错误** | 扫描或解析过程中发生错误 🔴 |
| `invalid` | **无效** | Playbook 文件格式不合规或引用的角色/文件不存在 ⚫ |

---

## 二、变量管理

Playbook 中声明的变量是执行时动态注入的参数。平台提供两种变量管理模式：

### 2.1 变量配置模式 (config_mode)

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `auto` | **自动** | 使用自动扫描发现的变量，无需手动配置 |
| `enhanced` | **增强** | 在自动扫描基础上，允许手动调整变量名、类型、默认值等 |

### 2.2 变量属性

每个变量包含以下属性：

| 属性 | 说明 | 示例值 |
|------|------|--------|
| **name** | 变量名称 | `target_host` |
| **type** | 数据类型 | `string` / `number` / `boolean` / `list` / `object` |
| **required** | 是否必填 | `true` / `false` |
| **default** | 默认值 | `localhost` |
| **description** | 变量描述 | `目标主机地址` |
| **source_file** | 变量定义所在文件 | `vars/main.yml` |

---

## 三、字段说明

| 字段 | 说明 | 示例值 |
|------|------|--------|
| **名称** | Playbook 的名称 | `disk-cleanup-playbook` |
| **描述** | 简短描述 | `清理磁盘空间的运维脚本` |
| **入口文件** (file_path) | YAML 入口文件路径 | `playbooks/disk-cleanup/main.yml` |
| **所属仓库** | 来源的 Git 仓库 | `ops-playbooks` |
| **状态** | 当前状态 | `ready` |
| **变量数** | 配置的变量数量 | `5` |
| **默认超时** (timeout) | 执行超时时间（分钟） | `60` |
| **标签** (tags) | 分类标签 | `清理`, `磁盘` |

## 四、核心操作

### 4.1 变量配置
1. 在 Playbook 列表中点击 **配置** 按钮。
2. 查看自动扫描到的变量列表。
3. 调整每个变量的：类型、是否必填、默认值、描述。
4. 确认配置后，Playbook 状态从 `pending` 变为 `ready`。

### 4.2 重新扫描
当仓库代码更新后，可手动触发重新扫描。平台会：
- 重新解析 Playbook 文件。
- 检测新增或删除的变量。
- 如果有变量变更，关联的任务模板会被标记为 `needs_review`（需要审核）。

### 4.3 查看源码
支持直接在平台内预览 Playbook 的 YAML 源文件内容，无需访问 Git 仓库。

### 4.4 查看扫描日志
每次扫描操作都会记录日志，包括：
- 扫描的文件数 (files_scanned)
- 发现的变量数 (variables_found)
- 新增 / 移除的变量数

---

## 五、变量变更检测

当 Playbook 的变量发生变化时，平台会自动检测并标记受影响的任务模板。这个机制确保：
- 新增的必填变量不会导致执行时参数缺失。
- 删除的变量不会导致配置引用失效。
- 管理员收到通知后可以及时更新任务模板的变量配置。

---

## 六、最佳实践

1. **规范化声明变量**：在 Playbook 的 `defaults/main.yml` 或 `vars/main.yml` 中显式声明所有变量及其默认值，便于平台自动扫描。
2. **使用标签分类**：给 Playbook 添加有意义的标签（如 `清理`、`修复`、`监控`），方便后续筛选。
3. **及时确认 pending 状态**：首次扫描后尽快完成变量配置，使 Playbook 可用于任务模板。
4. **关注变量变更通知**：仓库更新后检查是否有任务模板标记为 `needs_review`。
