# 自愈流程 (Healing Flows) 功能指南

![自愈流程编辑器](/guides/images/healing_flow_overview.webp)

自愈流程是 Auto Healing 平台的**核心编排引擎**。它提供了一个强大的 DAG（有向无环图）可视化拖拽编辑器，允许运维人员将多个操作步骤（主机提取、CMDB 校验、文件执行、人工审批、通知发送等）组装成一条复合型的自动修复链条。

---

## 一、节点类型 (node_type) 字典

流程由多种功能节点组成，通过连线定义执行顺序和分支逻辑：

| 枚举值 | 中文含义 | 颜色标识 | 说明 |
|--------|---------|---------|------|
| `start` | **开始节点** | 🟢 绿色 | 流程的入口点，每个流程有且仅有一个。接收工单数据作为初始上下文 |
| `end` | **结束节点** | 🔴 红色 | 流程的终止点。到达此节点表示流程执行完毕 |
| `host_extractor` | **主机提取** | 🔵 蓝色 | 从工单数据中提取受影响的主机标识（IP/主机名） |
| `cmdb_validator` | **CMDB 校验** | 🟤 棕色 | 验证提取的主机是否在 CMDB 中注册且状态正常（非维护中） |
| `execution` | **执行节点** | 🟣 紫色 | 调用任务模板执行 Ansible Playbook。支持成功/失败两条分支出口 |
| `approval` | **审批节点** | 🟠 橙色 | 暂停流程等待人工审批。审批通过后继续，拒绝后走拒绝分支 |
| `notification` | **通知节点** | 🔵 蓝色高亮 | 通过配置的渠道发送通知消息（邮件/钉钉/Webhook） |
| `condition` | **条件节点** | 🟡 黄色 | 基于表达式判断结果选择不同分支。如：执行次数 < 3 则重试 |
| `set_variable` | **变量赋值** | ⚪ 白色 | 在流程上下文中设置或更新变量值 |
| `compute` | **计算节点** | 🔷 青色 | 执行表达式计算（数学运算、字符串处理等），结果写入上下文 |

## 二、节点状态 (node_state) 字典

流程执行时，每个节点会经历以下状态：

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `pending` | **等待中** | 节点尚未开始执行，等待前置节点完成 |
| `running` | **执行中** | 节点正在执行中 |
| `success` | **成功** | 节点执行成功 ✅ |
| `partial` | **部分成功** | 仅 `execution` 节点 — 部分主机成功，部分失败 ⚠️ |
| `failed` | **失败** | 节点执行出错 ❌ |
| `skipped` | **已跳过** | 因分支条件不满足，该节点被跳过 ⏭ |
| `waiting_approval` | **等待审批** | 仅 `approval` 节点 — 流程暂停等待人工决策 🕐 |

---

## 三、连线与分支

节点之间通过**有向连线**（Edge）定义执行顺序。部分节点支持多出口（分支）：

### 3.1 执行节点 (execution) 的出口

| 出口标识 (sourceHandle) | 中文含义 | 何时走此分支 |
|------------------------|---------|-------------|
| `success` | **成功** | Playbook 在所有目标主机上执行成功 |
| `failed` | **失败** | 执行失败（部分或全部主机） |

### 3.2 审批节点 (approval) 的出口

| 出口标识 | 中文含义 | 何时走此分支 |
|---------|---------|-------------|
| `approved` | **已批准** | 审批人点击"通过" |
| `rejected` | **已拒绝** | 审批人点击"拒绝" |

### 3.3 条件节点 (condition) 的出口

| 出口标识 | 中文含义 | 何时走此分支 |
|---------|---------|-------------|
| `true` | **条件为真** | 表达式求值结果为 true |
| `false` | **条件为假** | 表达式求值结果为 false |

## 四、流程上下文 (Context)

流程实例运行时维护一个全局上下文（JSON 对象），各节点可以读写上下文中的变量：

- **start 节点**：自动注入工单数据（`incident` 对象）和匹配的规则信息。
- **host_extractor**：写入 `extracted_hosts`（提取到的主机列表）。
- **cmdb_validator**：写入 `validated_hosts`（校验通过的主机列表）。
- **execution**：写入执行结果（`execution_result`）。
- **set_variable / compute**：自定义写入任意变量。
- **condition**：读取上下文变量进行条件判断。

---

## 五、表达式引擎

条件节点和计算节点使用 `expr` 表达式引擎，支持：

| 类型 | 运算 | 示例 |
|------|------|------|
| **数学** | `+`, `-`, `*`, `/`, `%` | `retry_count + 1` |
| **比较** | `==`, `!=`, `>`, `<`, `>=`, `<=` | `retry_count < 3` |
| **逻辑** | `&&`, `\|\|`, `!` | `status == "failed" && retry_count < 3` |
| **字符串** | `contains`, `startsWith`, `endsWith` | `host contains "web"` |
| **数组** | `len`, `any`, `all`, `filter`, `map` | `len(failed_hosts) > 0` |

## 六、核心操作

### 6.1 创建流程
1. 点击 **+ 新建**，输入流程名称和描述。
2. 进入**可视化编辑器**。
3. 从左侧工具栏拖拽节点到画布。
4. 连线定义执行顺序和分支逻辑。
5. 双击节点配置其参数（如选择任务模板、设置审批人等）。
6. 保存流程。

### 6.2 编辑流程
在流程列表点击 **编辑** 重新进入编辑器修改。修改后的流程**不会影响正在运行的实例**——已启动的实例使用的是启动时的流程快照。

### 6.3 启用/停用
通过 `is_active` 开关控制流程是否可被自愈规则引用。停用的流程不会出现在规则的流程选择列表中。

### 6.4 试运行 (Dry Run)
提供**模拟运行**功能，可以用虚构的工单数据测试流程的执行路径，而不实际执行 Playbook。

## 七、最佳实践

1. **渐进设计**：先搭建简单的 `start → execution → end` 线性流程，验证通过后再添加条件分支和审批节点。
2. **总是添加通知节点**：在 `end` 节点前添加通知节点，确保流程结束后相关人员收到通知。
3. **错误处理**：为 execution 节点的 `failed` 出口设计降级逻辑（如重试、发送告警）。
4. **善用 CMDB 校验**：在执行之前添加 cmdb_validator，避免在维护中或已下线的主机上执行操作。
