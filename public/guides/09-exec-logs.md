# 执行记录 (Execution Logs) 功能指南

![执行记录列表](/guides/images/execution_logs_overview.webp)

执行记录是系统中所有任务执行的**不可篡改的流水线日志**。每次通过手动执行、定时调度、或自愈流程触发的 Playbook 运行，都会在此留下一条完整的执行记录，包括输入参数、执行过程日志、最终结果和耗时等详细信息。

---

## 一、执行状态 (status) 字典

| 枚举值 | 中文含义 | 颜色标识 | 说明 |
|--------|---------|---------|------|
| `pending` | **排队中** | ⚪ 灰色 | 任务已提交，等待执行引擎分配资源 |
| `running` | **执行中** | 🔵 蓝色 | Ansible 正在目标主机上运行 Playbook |
| `success` | **成功** | 🟢 绿色 | 所有目标主机上的 Playbook 全部执行成功 ✅ |
| `partial` | **部分成功** | 🟡 黄色 | 部分主机执行成功，部分主机失败 ⚠️ |
| `failed` | **失败** | 🔴 红色 | 所有主机均执行失败或出现致命错误 ❌ |
| `timeout` | **超时** | 🟠 橙色 | 执行时间超过模板配置的超时阈值 |
| `cancelled` | **已取消** | ⚫ 黑色 | 用户手动取消了正在执行的任务 |

---

## 二、触发来源 (trigger_type) 字典

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `manual` | **手动执行** | 用户在任务模板页面手动触发 |
| `schedule` | **定时执行** | 由定时任务自动触发 |
| `healing` | **自愈触发** | 由自愈流程的 execution 节点触发 |
| `api` | **API 调用** | 通过 REST API 接口触发 |

---

## 三、记录详情

### 3.1 概览信息

| 字段 | 说明 | 示例值 |
|------|------|--------|
| **执行 ID** | 唯一执行标识 | `run-20240115-001` |
| **任务模板** | 触发执行的模板名称 | `日志归档清理-应用服务器` |
| **Playbook** | 执行的 Playbook 文件 | `cleanup/main.yml` |
| **触发来源** | 执行是如何被触发的 | `manual` |
| **目标主机** | 本次执行涉及的主机列表 | `10.0.1.1, 10.0.1.2` |
| **开始时间** | 执行开始的时间戳 | `2024-01-15 10:30:15` |
| **结束时间** | 执行完成的时间戳 | `2024-01-15 10:32:45` |
| **耗时** | 总执行时间 | `2 分 30 秒` |
| **状态** | 最终执行结果 | `success` |

### 3.2 主机级执行结果

对于每台目标主机，记录详细的 Ansible 执行摘要：

| 指标 | 说明 |
|------|------|
| **ok** | 成功执行的任务步骤数 |
| **changed** | 发生了实际变更的步骤数 |
| **unreachable** | 主机不可达的次数 |
| **failed** | 执行失败的步骤数 |
| **skipped** | 通过条件判断跳过的步骤数 |
| **rescued** | 通过 rescue 恢复的步骤数 |
| **ignored** | 标记为忽略错误的步骤数 |

### 3.3 控制台日志 (Console Output)
展示 Ansible 的完整标准输出，包括每个 task 的执行详情、变更内容和错误信息。支持：
- 按关键词搜索日志内容
- 按日志级别过滤（INFO / WARN / ERROR）
- 复制日志到剪贴板

---

## 四、核心操作

### 4.1 查看执行详情
点击执行记录可展开详情面板，查看完整的执行过程数据。

### 4.2 重新执行
对于失败的任务，可点击 **重新执行** 按钮以相同参数再次触发。

### 4.3 取消执行
对于正在运行中的任务，可点击 **取消** 按钮终止执行。

### 4.4 筛选与搜索
- 按状态筛选（成功/失败/部分成功等）
- 按模板名称搜索
- 按触发来源过滤
- 按时间范围查询

---

## 五、最佳实践

1. **定期排查 `partial` 状态**：部分成功意味着有主机存在问题，需要排查具体原因。
2. **利用控制台日志排错**：执行失败时，控制台会输出 Ansible 的详细报错信息。
3. **监控执行耗时**：如果某个模板的执行时间持续增长，可能说明目标系统负载异常。
