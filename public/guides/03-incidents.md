# 工单管理 (Incidents) 功能指南

![工单管理列表](/guides/images/incidents_overview.webp)

工单管理是告警事件的**统一归集与生命周期管理中心**。外部监控系统（Zabbix、Prometheus、ServiceNow 等）通过插件推送的告警数据，以及平台内部产生的事件，都会以"工单"的形式在此汇聚。自愈引擎正是以工单为输入，扫描并匹配规则后自动触发修复流程。

---

## 一、工单来源

工单的来源取决于您安装的插件类型：

| 来源方式 | 说明 | 典型场景 |
|---------|------|---------|
| **插件定时轮询 (Polling)** | 插件按配置的间隔（如每 5 分钟）从外部系统 API 拉取新告警 | Zabbix、ServiceNow 等有 REST API 的系统 |
| **Webhook 推送** | 外部系统主动向平台 Webhook 端点推送告警数据 | Prometheus Alertmanager、PagerDuty |
| **手动创建** | 运维人员在平台内手动新增工单 | 巡检发现问题、电话报修 |

## 二、字段详解

### 2.1 工单基本字段

| 字段 | 类型 | 说明 | 示例值 |
|------|------|------|--------|
| **工单标题** (title) | 文本 | 告警的简短描述 | `MySQL 连接池耗尽` |
| **外部 ID** (external_id) | 文本 | 来源系统中的原始工单编号 | `INC00012345` |
| **来源** (source_plugin_name) | 文本 | 产生该工单的插件名称 | `Zabbix-Prod` |
| **描述** (description) | 长文本 | 告警的详细描述信息 | `Connection pool exhausted...` |
| **分类** (category) | 文本 | 事件分类标签 | `网络` / `数据库` / `应用` |
| **影响 CI** (affected_ci) | 文本 | 受影响的配置项标识 | `db-master-01` |
| **影响服务** (affected_service) | 文本 | 受影响的业务服务 | `订单服务` |
| **报告人** (reporter) | 文本 | 告警的报告人/系统 | `monitoring-system` |
| **处理人** (assignee) | 文本 | 被分配处理该工单的人员 | `ops-zhangsan` |
| **原始数据** (raw_data) | JSON | 来源系统推送的完整原始 JSON 数据 | `{"host":"10.0.1.5","trigger":...}` |

### 2.2 工单状态 (status) 字典

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `open` | **待处理** | 工单刚创建或被重新打开，尚未开始处理 |
| `in_progress` | **处理中** | 工单已被分配或自愈引擎正在处理中 |
| `resolved` | **已解决** | 问题已被修复（手动或自动），等待确认关闭 |
| `closed` | **已关闭** | 工单已完结，不再需要任何操作 |

### 2.3 告警级别 (severity) 字典

| 枚举值 | 中文含义 | 颜色标识 | 说明 |
|--------|---------|---------|------|
| `critical` | **严重** | 🔴 红色 | 影响核心业务，需立即处理 |
| `high` | **高** | 🟠 橙色 | 高优先级，尽快修复 |
| `medium` | **中** | 🟡 黄色 | 一般优先级，计划修复 |
| `low` | **低** | ⚪ 灰色 | 低优先级，可择机处理 |

### 2.4 自愈状态 (healing_status) 字典

这是整个自愈平台最核心的状态字段之一，标记了每个工单在自愈引擎中的处理情况：

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `pending` | **待处理** | 工单刚入库，尚未被自愈引擎扫描或无匹配规则正在等待处理 |
| `processing` | **处理中** | 已匹配到自愈规则，正在执行自愈流程 |
| `healed` | **已自愈** | 自愈流程成功执行完成，问题已自动修复 ✅ |
| `failed` | **自愈失败** | 自愈流程执行过程中发生错误，需要人工介入 ❌ |
| `skipped` | **已跳过** | 自愈引擎扫描后未找到匹配的规则，工单被跳过 ⏭ |
| `dismissed` | **已忽略** | 运维人员手动标记为忽略，不需要自愈处理 🚫 |

> **提示：** 在工单列表页面，可以通过筛选栏按自愈状态快速过滤，例如只查看"自愈失败"的工单以便排查问题。

## 三、核心操作

### 3.1 查看工单详情
点击工单标题可打开详情面板，查看完整信息、原始数据 JSON、以及自愈执行关联信息。

### 3.2 手动变更状态
在详情页或列表操作栏中，支持手动修改工单状态和自愈状态。场景例子：
- 误报的告警：将自愈状态设为 `dismissed`（已忽略）
- 需要手动修复：将工单状态设为 `in_progress`（处理中）

### 3.3 手动触发自愈
对于 `trigger_mode = manual` 的规则匹配到的工单，需要运维人员在**待办中心**手动确认后才会触发执行。

### 3.4 筛选与搜索
- **关键词搜索**：按标题、外部 ID、影响 CI 等搜索
- **高级筛选**：按来源插件、告警级别、工单状态、自愈状态组合筛选
- **时间范围**：按创建时间筛选

### 3.5 添加备注
可在工单详情中追加人工备注，记录排查过程和处理结果。

## 四、与自愈引擎的联动

工单是自愈引擎的**核心输入**。完整的联动流程如下：

```
外部告警 → 插件接收 → 创建工单(healing_status=pending) 
→ 自愈引擎扫描 → 匹配规则 
  ├─ 无匹配 → healing_status=skipped
  ├─ 匹配(auto) → healing_status=processing → 执行流程
  │   ├─ 成功 → healing_status=healed
  │   └─ 失败 → healing_status=failed
  └─ 匹配(manual) → 进入待办中心等待审批
```

---

## 五、最佳实践

1. **关注 `failed` 工单**：自愈失败的工单需要定期巡检，分析失败原因并改进 Playbook。
2. **善用 `dismissed`**：对于已知的误报或不需自愈的告警，及时标记忽略避免干扰统计。
3. **原始数据利用**：工单的 `raw_data` 字段保存了来源系统的完整数据，是排查问题的第一手资料。
4. **设置合理的告警级别映射**：在插件的字段映射中，确保 severity 正确映射，直接影响待办排序。
