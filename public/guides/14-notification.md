# 通知管理功能指南

![通知渠道列表](/guides/images/notification_channels.webp)

通知管理模块提供**多渠道消息推送**能力，支持在自愈流程执行、任务完成、审批请求等场景自动发送通知。由三个子功能组成：**通知渠道**（定义"通过什么发"）、**通知模板**（定义"发什么内容"）、**通知日志**（记录"发了什么"）。

---

## 一、通知渠道 (Channels)

### 1.1 渠道类型

| 类型 | 代码标识 | 说明 | 配置要点 |
|------|---------|------|---------|
| **邮件** | `email` | 通过 SMTP 服务器发送邮件 | SMTP 地址、端口、发件人、认证 |
| **钉钉** | `dingtalk` | 通过钉钉群机器人 Webhook 推送 | Webhook URL、签名密钥 |
| **Webhook** | `webhook` | 通用 HTTP POST 推送 | 目标 URL、Header、Basic Auth |

### 1.2 渠道字段

| 字段 | 说明 | 示例值 |
|------|------|--------|
| **名称** | 渠道名称 | `OPS钉钉群` |
| **类型** | 渠道类型 | `dingtalk` |
| **状态** | 是否启用 | `active` / `inactive` |
| **配置** | 连接配置（JSON） | `{"webhook_url":"https://..."}` |

---

## 二、通知模板 (Templates)

### 2.1 模板变量

模板内容支持使用 `{{ }}` 语法插入动态变量：

| 变量 | 说明 | 示例值 |
|------|------|--------|
| `{{ .Title }}` | 通知标题 | `自愈流程执行成功` |
| `{{ .Content }}` | 通知正文 | `磁盘清理已完成` |
| `{{ .IncidentTitle }}` | 关联工单标题 | `磁盘使用率超过90%` |
| `{{ .FlowName }}` | 流程名称 | `磁盘空间自愈流程` |
| `{{ .Status }}` | 执行状态 | `success` |
| `{{ .Host }}` | 目标主机 | `10.0.1.1` |
| `{{ .Time }}` | 事件时间 | `2024-01-15 10:30:00` |

### 2.2 模板字段

| 字段 | 说明 |
|------|------|
| **名称** | 模板名称 |
| **类型** | 适用的通知渠道类型 |
| **标题模板** | 通知标题的模板文本 |
| **内容模板** | 通知正文的模板文本（支持 Markdown 格式） |

## 三、通知日志 (Logs)

### 3.1 日志状态字典

| 枚举值 | 中文含义 | 说明 |
|--------|---------|------|
| `sent` | **已发送** | 通知成功送达目标渠道 ✅ |
| `failed` | **发送失败** | 发送过程出错（如 Webhook 超时、SMTP 认证失败） ❌ |

### 3.2 日志字段

| 字段 | 说明 |
|------|------|
| **渠道** | 使用的通知渠道 |
| **模板** | 使用的通知模板 |
| **状态** | 发送结果 |
| **收件人** | 通知接收方 |
| **发送时间** | 发送时间戳 |
| **错误信息** | 失败时的错误消息 |

---

## 四、重试机制

通知发送失败时，平台会自动执行**顺序退避重试**：
- 第 1 次重试：等待 5 秒
- 第 2 次重试：等待 15 秒
- 第 3 次重试：等待 30 秒
- 超过 3 次后放弃并记录最终失败状态

---

## 五、最佳实践

1. **多渠道备份**：为重要通知同时配置邮件和钉钉渠道，确保不遗漏。
2. **模板复用**：创建通用的成功/失败通知模板，在多个流程中复用。
3. **定期检查日志**：关注失败频率高的渠道，可能是配置过期或目标服务不可用。
4. **Webhook 安全**：使用 Basic Auth 保护 Webhook 端点，防止未授权调用。
