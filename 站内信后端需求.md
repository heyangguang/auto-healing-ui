# 站内信（Site Messages）后端需求

> **重要说明**：这是「站内信」模块，不是已有的通知管理（Notification）模块。  
> 站内信是平台内部给用户推送的消息，和通知渠道（邮件/钉钉/Webhook）、通知模板、通知发送记录**完全无关**，是独立的功能。

---

## 一、功能概述

系统需要一个「站内信」功能，用于向用户发送平台内的系统消息。  
用户可以在站内信页面查看所有消息、搜索消息、标记已读。  
导航栏的铃铛图标会实时显示未读消息数量。

---

## 二、核心业务需求

### 1. 消息存储

每条站内信需要包含以下信息：

| 字段 | 说明 |
|------|------|
| 分类 | 消息类别，如"产品消息"、"服务消息"、"活动通知"、"故障通知"等 |
| 标题 | 消息标题，纯文本 |
| 内容 | 消息正文，支持 HTML 富文本 |
| 发送时间 | 消息创建/发送的时间 |
| 已读状态 | 该消息是否已被用户阅读（**用户维度**） |

### 2. 消息查询

- 分页查询所有站内信，支持 `page` 和 `page_size` 分页参数
- 支持按**标题关键词**搜索（模糊匹配）
- 支持按**分类**筛选
- 返回结果按**时间倒序**排列（最新的在前）
- 返回总数 `total`，前端用于分页和铃铛显示未读数量

### 3. 未读数量

- 获取当前用户的未读站内信总数
- **注意**：前端铃铛每 60 秒轮询一次，这个接口需要**快速响应**，尽量轻量

### 4. 标记已读

- **批量标记已读**：传入一组消息 ID，将这些消息标记为已读
- **全部标记已读**：将当前用户的所有未读消息一次性标记为已读

### 5. 消息发送能力

- 需要有创建站内信的能力，至少支持系统内部调用（如某个事件触发后自动生成一条站内信）
- 消息发送时需要指定：分类、标题、内容
- 可以考虑支持：
  - 向**所有用户**广播
  - 向**指定用户/角色**发送

---

## 三、其他要求

1. **与通知管理模块完全独立**：站内信是独立的数据表和接口，不要复用现有的 `notification_logs` / `notification_templates` / `notification_channels` 等表
2. **多用户已读状态**：同一条广播消息，A 用户已读不影响 B 用户的已读状态
3. **种子数据**：后端完成后，需要有 10-20 条不同分类的初始测试数据

---

## 四、前端数据结构参考

前端目前 mock 的数据结构（仅供参考，后端可以按自己的方式设计，前端会适配）：

```json
{
  "id": "唯一标识",
  "category": "产品消息-产品的创建、开通信息通知",
  "title": "阿里云产品动态月刊",
  "content": "<h2>标题</h2><p>HTML富文本内容</p>",
  "date": "2026-02-17",
  "time": "14:30:00",
  "isRead": false
}
```

前端目前调用的操作：

1. **查询消息列表**（分页 + 搜索 + 分类筛选）
2. **获取未读数量**
3. **批量标记已读**（传入 ID 数组）
4. **全部标记已读**
